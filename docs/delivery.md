# Поставка движка: Docker-образ и обновления

Инструкция «с нуля»: как отдать cortex компании, не передавая исходники, и как
выкатывать обновления. Если процесс в новинку — читай по порядку.

## Идея в двух словах
- **Код (этот репозиторий) остаётся у тебя.** Ты его не отдаёшь.
- Ты собираешь из кода **образ** (Docker image) — самодостаточный «архив с программой».
- Компания **запускает образ** у себя (свой сервер, своя БД, свои ключи Yandex).
- Изменила движок → собрала **новый образ с новой версией** → компания **переразвернула** его.

## Мини-глоссарий
| Термин | Простыми словами |
|---|---|
| **Образ (image)** | замороженная упаковка программы + её зависимостей. Неизменна. Имеет имя и версию-тег. |
| **Тег** | версия образа, напр. `cortex:1.0.0`. Новый билд = новый тег. |
| **Реестр (registry)** | «склад образов». У компании — Yandex Container Registry (`cr.yandex/...`). |
| **Контейнер** | образ, **запущенный** как процесс. Это и есть «инстанс». |
| **Деплой / редеплой** | запустить (перезапустить) контейнер из нужного тега. |

> Код → образ → (push в реестр) → компания запускает контейнер. Контейнер = инстанс.

## Шаг 1. Подготовка (один раз)
В репозитории уже есть:
- `Dockerfile` — рецепт сборки образа.
- `.dockerignore` — что НЕ класть в образ (node_modules, `.env`, `.git`).

Проверь, что образ вообще собирается локально:
```bash
docker build -t cortex:dev .
# запустить локально на своём .env, проверить здоровье:
docker run --rm --env-file .env -p 8080:8080 cortex:dev
curl -s localhost:8080/health
```

## Шаг 2. Собрать и опубликовать версию
```bash
# 1) тег с версией + адрес реестра компании (его id даст их DevOps)
IMAGE=cr.yandex/<REGISTRY_ID>/cortex
VER=1.0.0

# 2) залогиниться в их реестр (один раз; команду/доступ даёт DevOps)
yc container registry configure-docker      # для Yandex CR

# 3) собрать под нужную платформу и запушить
docker build --platform linux/amd64 -t $IMAGE:$VER .
docker push $IMAGE:$VER
```
Готово — образ `cortex:1.0.0` лежит в их реестре. Исходники туда не попадают.

### Быстрый путь — Makefile (как у сервисов компании)
Чтобы не печатать команды руками, в репозитории есть `Makefile`. Один раз добавь в `.env`:
```
IMAGE_NAME=cr.yandex/<REGISTRY_ID>/cortex
```
Дальше:
```bash
make release v=1.0.0     # собрать образ и запушить в реестр
make migrate v=1.0.0     # прогнать миграции на БД из .env (если менялась схема)
make run-local           # локально проверить, что образ собирается и стартует
```
`make release` = `build` + `push`. Версию указываешь через `v=` (без неё Makefile подскажет).

## Шаг 3. Что отдать компании при ПЕРВОМ развёртывании
1. **Имя образа+тег** (`cr.yandex/<id>/cortex:1.0.0`).
2. **Спецификацию env** (что заполнить их значениями) — `.env.example` подходит:
   - `DATABASE_URL` (их Postgres), `SERVER_ADDRESS`,
   - `YANDEX_API_KEY`, `YANDEX_FOLDER_ID`, `AI_PROVIDER=yandex`,
   - опц. `ENABLE_HEADLESS`, тарифы `YANDEX_*_RUB_PER_1K`.
3. **Команда миграций** (создать таблицы в их БД, один раз и при изменении схемы):
   ```bash
   docker run --rm --env-file .env $IMAGE:$VER npm run db:migrate
   ```
4. **Команда запуска** (их оркестратор/compose): контейнер из образа, проброс env, порт 8080, healthcheck `GET /health`.
5. **API-ключ** для их бэкенда — выдаёшь отдельно (секретный `sk_`, см. [quotcat-poc.md](quotcat-poc.md)).

## Цикл обновления (главное, что ты спрашивала)
Когда вносишь изменения в движок:

```
правки в коде → новый тег (1.1.0) → docker build → docker push → компания pull+redeploy
```
1. Собрать и запушить новый тег: `VER=1.1.0; docker build ... ; docker push $IMAGE:$VER`.
2. Компания меняет тег в своём деплое на `1.1.0` и перезапускает контейнер (обычно zero-downtime).
3. **Если менялась схема БД** — перед/во время выката прогнать `npm run db:migrate` на их БД (см. выше). Если схему не трогала — миграции не нужны, просто новый образ.
4. **Контракт `/api/v1` не меняли → код Quotcat трогать не нужно** — они получают «движок получше». Ломающее изменение API = заранее согласовать (`/v1` → `/v2`).
5. Откат: вернуть предыдущий тег (`1.0.0`) и перезапустить.

## Вариант с headless (парсинг JS-страниц)
Базовый образ — без браузера (легче). Если нужен `ENABLE_HEADLESS=1`:
- собирать от образа с браузером, напр. `mcr.microsoft.com/playwright:v1.61.1-jammy`, и в Dockerfile добавить `RUN npx playwright install --with-deps chromium`, а в `npm install` убрать `--omit=optional`;
- либо держать **два образа**: `cortex` (лёгкий) и `cortex-headless`. Решается при необходимости.

## Чего НЕ делать
- ❌ Не класть `.env`/ключи внутрь образа (они снаружи; `.dockerignore` уже исключает `.env`).
- ❌ Не давать компании доступ к репозиторию-источнику — только образ.
- ⚠️ Образ Node содержит JS — он защищает от «скопировали/форкнули», но не от упорного реверса. Реальная защита IP — **лицензия и договор**, не образ.
